# Yuseong Save 사장님 웹 콘솔 설계

## 1. 웹 프론트의 역할

앱은 소비자가 주변 가게의 할인, 행사, 재고, 좌석 정보를 실시간으로 확인하는 화면이다.  
웹은 소상공인 사장님이 앱에 노출될 가게 정보를 직접 등록하고 관리하는 운영 콘솔이다.

즉, 웹의 핵심 목적은 다음 4가지다.

1. 내 가게 기본 정보 등록 및 수정
2. 실시간 할인, 타임세일, 행사, 공지 등록
3. 테이블, 재고, 영업상태 같은 실시간 운영 정보 관리
4. 앱에서 어떻게 보이는지 미리 확인

---

## 2. 사용자 구분

### 사장님

- 자신의 가게를 등록한다.
- 할인, 행사, 신메뉴, 포스터를 등록한다.
- 현재 빈 테이블 수, 대기 인원, 재고 상태를 수정한다.
- 앱에 노출되는 내 가게 정보를 확인한다.

### 관리자

해커톤 MVP에서는 필수 범위에서 제외한다.  
다만 추후 확장을 위해 사업자 승인, 신고 관리, 공공데이터 연동 검수 기능을 별도 관리자 페이지로 분리할 수 있다.

## 4. MVP 우선순위

### 1순위: 반드시 구현

| 화면 | 목적 |
| --- | --- |
| 로그인 / 회원가입 | 사장님 계정 진입 |
| 가게 등록 | 앱에 노출될 가게 기본 정보 입력 |
| 대시보드 | 현재 운영 상태를 한눈에 확인 |
| 할인/타임세일 등록 | 서비스 차별점인 실시간 혜택 등록 |
| 실시간 운영 정보 수정 | 테이블, 대기, 재고 상태 빠른 수정 |
| 앱 미리보기 | 사장님이 입력한 정보가 앱에서 어떻게 보이는지 확인 |

### 2순위: 가능하면 구현

| 화면 | 목적 |
| --- | --- |
| 홍보 포스터 관리 | 앱의 프로모션 탭에 노출 |
| 행사/공지 목록 관리 | 등록한 홍보글 수정/삭제 |
| 통계 요약 | 조회수, 클릭수, 진행 중 혜택 수 확인 |

### 3순위: 추후 확장

| 화면 | 목적 |
| --- | --- |
| 사업자 인증 | 실제 운영 가게 검증 |
| 리뷰/문의 관리 | 소비자 반응 관리 |
| 알림 발송 관리 | 주변 사용자 대상 푸시 알림 |

## 5. 화면별 상세 설계

## 5.1 로그인 화면

### 목적

사장님이 웹 콘솔에 진입한다.

### 주요 UI

- 서비스 로고: `Yuseong Save`
- 설명 문구: `우리 가게의 실시간 혜택을 동네 손님에게 알려보세요`
- 아이디 입력
- 비밀번호 입력
- 로그인 버튼
- 회원가입 링크

### 기능

- 아이디/비밀번호 로그인
- 로그인 성공 시 대시보드 이동
- 가게 미등록 계정은 가게 등록 화면으로 이동

### 예외 처리

- 아이디 또는 비밀번호 미입력
- 로그인 실패
- 등록된 가게가 없는 계정

---

## 5.2 회원가입 화면

### 목적

사장님 계정을 생성한다.

### 주요 UI

- 이름
- 휴대폰 번호
- 아이디
- 비밀번호
- 비밀번호 확인
- 회원가입 버튼

### 기능

- 계정 생성
- 가입 완료 후 가게 등록 화면 이동

### 예외 처리

- 중복 아이디
- 비밀번호 불일치
- 필수값 미입력

---

### 주요 UI

- 가게명
- 한 줄 소개
- 상세 설명
- 카테고리 선택
  - 음식점
  - 카페
  - 마트
  - 베이커리
  - 식료품
  - 기타
- 주소 입력
- 지도 위치 선택
- 대표 이미지 업로드
- 영업 시간 입력
- 전화번호
- 저장 버튼

### 기능

- 가게 기본 정보 등록
- 이미지 업로드
- 주소 기반 위치 등록
- 앱 지도/목록/상세 페이지 노출 데이터 생성

### 예외 처리

- 필수 정보 누락
- 이미지 용량 초과
- 주소 검색 실패
- 위치 좌표 미선택

  - 테이블 상태 수정

### 기능

- 현재 영업 상태 확인
- 진행 중인 이벤트 확인
- 운영 정보 빠른 수정
- 앱 미리보기 이동

## 5.5 실시간 운영 관리 화면

### 목적

소비자 앱에 실시간으로 보여줄 운영 상태를 빠르게 수정한다.

### 주요 UI

- 영업 상태 토글
  - 영업중
  - 브레이크타임
  - 마감
- 빈 테이블 수 입력
- 대기 인원 입력
- 재고 상태 선택
  - 충분
  - 보통
  - 얼마 없음
  - 품절
- 마지막 업데이트 시간
- 저장 버튼

### 기능

- 앱 상세 정보의 `이용 가능한 테이블` 영역 반영
- 앱 지도/목록 카드의 상태 배지 반영
- 재고 상태 업데이트

### 예외 처리

- 음수 입력 방지
- 숫자 외 입력 방지
- 너무 오래 업데이트하지 않은 정보는 `업데이트 필요` 표시

---

## 5.6 할인/타임세일 관리 화면

### 목적

서비스의 핵심 차별점인 실시간 할인과 타임세일을 등록한다.

### 주요 UI

- 혜택 유형 선택
  - 타임세일
  - 일반 할인
  - 쿠폰
  - 행사
- 제목
- 설명
- 할인율 또는 할인 금액
- 대상 상품
- 시작 시간
- 종료 시간
- 대표 이미지
- 앱 노출 위치 선택
  - 지도 배지
  - 목록 카드
  - 상세 페이지
  - 홍보 탭
- 등록 버튼

### 기능

- 할인/행사 등록
- 진행 전/진행 중/종료 상태 자동 분류
- 종료 시간이 지나면 자동 종료 처리
- 앱 화면에서 특별 배지로 노출

### 예외 처리

- 종료 시간이 시작 시간보다 빠른 경우
- 할인율이 0% 이하 또는 100% 초과인 경우
- 필수값 미입력
- 진행 중인 타임세일 중복 등록

### 기능

- 포스터형 홍보 콘텐츠 등록
- 앱 홍보 탭 리스트에 노출
- 포스터별 노출 상태 관리

### 예외 처리

- 이미지 미등록

## 5.8 앱 노출 미리보기 화면

### 목적

사장님이 입력한 정보가 소비자 앱에서 어떻게 보이는지 확인한다.

### 주요 UI

- 지도 카드 미리보기
- 목록 카드 미리보기
- 상세 페이지 미리보기
- 홍보 포스터 미리보기

### 기능

- 실제 앱 디자인과 유사한 모바일 프레임 제공
- 등록한 혜택 배지 확인
- 누락된 정보 표시

### 중요 포인트

앱 디자인과 흐름을 맞추기 위해 미리보기는 모바일 카드 형태로 제공한다.  
웹 자체는 데스크톱 콘솔이지만, 결과물은 앱 화면 기준으로 보여줘야 한다.

## 6. 디자인 방향

## 6.1 앱 디자인과 맞출 요소

첨부된 앱 시안의 핵심 톤은 다음과 같다.

- 따뜻한 오렌지/레드 계열 포인트 컬러
- 둥근 카드형 UI
- 연한 베이지/화이트 배경
- 배지 중심의 실시간 정보 강조
- 지도, 목록, 상세 화면 모두 `혜택`을 시각적으로 강조

웹도 이 흐름을 유지하되, 소비자용 앱보다 더 정리된 관리형 화면이어야 한다.

## 6.3 컴포넌트 스타일

### 버튼

- Primary 버튼: 진한 오렌지 배경, 흰색 텍스트
- Secondary 버튼: 흰색 배경, 오렌지 테두리
- Danger 버튼: 삭제/종료 기능에만 사용

### 카드

- `border-radius: 18px`
- 얇은 베이지 테두리
- 약한 그림자
- 내부 여백은 넉넉하게

### 배지

- `진행중`
- `타임세일`
- `마감 세일`
- `예약 가능`
- `재고 부족`
- `업데이트 필요`

앱에서 혜택 배지가 중요한 만큼, 웹에서도 같은 단어를 사용해야 한다.

### 입력 폼

- 라벨은 명확하게 표시
- 필수 항목은 `*` 표시
- 저장 전 미리보기 제공
- 모바일 앱에 노출되는 필드는 별도 안내 문구 제공

## 12. 최종 추천 구현 범위

해커톤 기준으로는 아래 순서가 가장 좋다.

1. 로그인/회원가입은 목업 또는 간단 구현
2. 가게 등록/수정 폼 구현
3. 대시보드 구현
4. 실시간 운영 상태 수정 구현
5. 타임세일/할인 등록 구현
6. 앱 미리보기 구현

가장 중요한 포인트는 `사장님이 입력한 정보가 앱에 바로 반영된다`는 경험이다.  
따라서 예쁜 관리자 페이지보다, 빠르게 등록하고 바로 미리볼 수 있는 흐름을 우선해야 한다.
